Лекція 10. Експлуатація КСЗІ: супровід, інциденти, модернізація

Управління конфігураціями, SOC, реагування на інциденти та цифрова криміналістика

1. Експлуатація КСЗІ

Після успішного впровадження та приймальних випробувань КСЗІ переходить у фазу постійної експлуатації. Цей етап є найтривалішим у життєвому циклі системи захисту та вимагає систематичного підходу до підтримки працездатності, оновлення компонентів та адаптації до нових загроз. Ефективна експлуатація КСЗІ базується на трьох стовпах: операційні процедури, автоматизація управління конфігураціями та управління оновленнями.

1.1. Операційні процедури безпеки

Операційні процедури безпеки (Security Operations Procedures) — це задокументовані регламенти повсякденних дій персоналу, що забезпечують безперебійне функціонування КСЗІ. Вони поділяються на щоденні, щотижневі та щомісячні.

Щоденні перевірки (Daily Checks)

  • Моніторинг алертів SIEM: перегляд критичних та високопріоритетних алертів за минулу добу, класифікація подій (true positive / false positive), ескалація підтверджених інцидентів
  • Перевірка стану компонентів: моніторинг доступності міжмережевих екранів, IDS/IPS, серверів SIEM, VPN-шлюзів; перевірка завантаженості ресурсів (CPU, RAM, дисковий простір)
  • Аналіз журналів (Log Review): перегляд логів автентифікації (невдалі спроби входу, блокування облікових записів), журналів міжмережевого екрану (заблоковані з'єднання), логів антивірусу (виявлені загрози)
  • Перевірка резервних копій (Backup Verification): підтвердження успішного завершення щоденних бекапів, перевірка цілісності резервних копій (checksums), контроль обсягу зайнятого сховища
  • Оновлення сигнатур: перевірка актуальності баз сигнатур IDS/IPS, антивірусного ПЗ, фідів threat intelligence

Щотижневі перевірки

  • Огляд звітів про сканування вразливостей (якщо налаштоване регулярне сканування)
  • Перевірка термінів дії сертифікатів SSL/TLS
  • Аналіз трендів: порівняння кількості подій безпеки з попереднім тижнем
  • Перегляд змін у конфігураціях (change review)
  • Тестове відновлення з резервної копії (вибірково)

Щомісячні перевірки

  • Повний аудит облікових записів: деактивація невикористовуваних, перевірка привілеїв
  • Огляд та оновлення правил міжмережевого екрану (видалення застарілих правил)
  • Перевірка відповідності конфігурацій базовим стандартам (CIS Benchmarks)
  • Підготовка місячного звіту з безпеки для керівництва
Практична порада Усі операційні процедури повинні бути задокументовані у вигляді чек-листів (runbooks) з чітким визначенням відповідальних осіб, термінів виконання та дій у випадку виявлення аномалій. Це забезпечує наступність при зміні персоналу та стандартизацію процесів.

1.2. IaC (Infrastructure as Code) для безпеки

Infrastructure as Code (IaC) — підхід до управління ІТ-інфраструктурою, при якому конфігурації описуються у вигляді коду, що зберігається у системі контролю версій (Git). IaC забезпечує повторюваність, документованість та аудит змін конфігурацій, що є критично важливим для безпеки.

Ansible: автоматизація налаштування безпеки

Ansible — система управління конфігураціями з відкритим вихідним кодом, що використовує безагентну архітектуру (підключення через SSH) та описує конфігурації у форматі YAML (playbooks). Ansible дозволяє автоматизувати типові задачі забезпечення безпеки: встановлення оновлень, налаштування файрволу, конфігурацію аудиту.

# security_hardening.yml — Ansible playbook для базового харденінгу Linux
---
- name: Security Hardening Playbook
  hosts: all
  become: yes   # виконання з правами root

  tasks:
    # 1. Оновлення всіх пакетів до останніх версій
    - name: Update all packages to latest version
      apt:
        upgrade: dist
        update_cache: yes
        cache_valid_time: 3600
      when: ansible_os_family == "Debian"

    - name: Update all packages (RHEL/CentOS)
      yum:
        name: '*'
        state: latest
        security: yes
      when: ansible_os_family == "RedHat"

    # 2. Налаштування файрволу (UFW для Ubuntu/Debian)
    - name: Install UFW firewall
      apt:
        name: ufw
        state: present

    - name: Set default UFW policy - deny incoming
      ufw:
        direction: incoming
        policy: deny

    - name: Set default UFW policy - allow outgoing
      ufw:
        direction: outgoing
        policy: allow

    - name: Allow SSH (port 22)
      ufw:
        rule: allow
        port: '22'
        proto: tcp

    - name: Allow HTTPS (port 443)
      ufw:
        rule: allow
        port: '443'
        proto: tcp

    - name: Enable UFW
      ufw:
        state: enabled

    # 3. Налаштування аудиту (auditd)
    - name: Install auditd
      apt:
        name: auditd
        state: present

    - name: Configure audit rules for sensitive files
      copy:
        content: |
          # Моніторинг змін у файлах автентифікації
          -w /etc/passwd -p wa -k identity
          -w /etc/shadow -p wa -k identity
          -w /etc/group -p wa -k identity
          -w /etc/gshadow -p wa -k identity
          -w /etc/sudoers -p wa -k sudoers
          # Моніторинг завантаження модулів ядра
          -w /sbin/insmod -p x -k modules
          -w /sbin/rmmod -p x -k modules
          -w /sbin/modprobe -p x -k modules
          # Моніторинг мережевих налаштувань
          -a always,exit -F arch=b64 -S sethostname -S setdomainname -k system-locale
          -w /etc/hosts -p wa -k system-locale
          -w /etc/sysconfig/network -p wa -k system-locale
        dest: /etc/audit/rules.d/security.rules
        owner: root
        group: root
        mode: '0640'
      notify: restart auditd

    # 4. Конфігурація SSH (харденінг)
    - name: Harden SSH configuration
      lineinfile:
        path: /etc/ssh/sshd_config
        regexp: "{{ item.regexp }}"
        line: "{{ item.line }}"
      loop:
        - { regexp: '^#?PermitRootLogin', line: 'PermitRootLogin no' }
        - { regexp: '^#?PasswordAuthentication', line: 'PasswordAuthentication no' }
        - { regexp: '^#?X11Forwarding', line: 'X11Forwarding no' }
        - { regexp: '^#?MaxAuthTries', line: 'MaxAuthTries 3' }
        - { regexp: '^#?Protocol', line: 'Protocol 2' }
        - { regexp: '^#?ClientAliveInterval', line: 'ClientAliveInterval 300' }
        - { regexp: '^#?ClientAliveCountMax', line: 'ClientAliveCountMax 2' }
      notify: restart sshd

    # 5. Налаштування політики паролів
    - name: Configure password policy (PAM)
      apt:
        name: libpam-pwquality
        state: present

    - name: Set password complexity requirements
      lineinfile:
        path: /etc/security/pwquality.conf
        regexp: "{{ item.regexp }}"
        line: "{{ item.line }}"
      loop:
        - { regexp: '^#?minlen', line: 'minlen = 12' }
        - { regexp: '^#?dcredit', line: 'dcredit = -1' }
        - { regexp: '^#?ucredit', line: 'ucredit = -1' }
        - { regexp: '^#?lcredit', line: 'lcredit = -1' }
        - { regexp: '^#?ocredit', line: 'ocredit = -1' }

  handlers:
    - name: restart auditd
      service:
        name: auditd
        state: restarted

    - name: restart sshd
      service:
        name: sshd
        state: restarted
# Запуск playbook
ansible-playbook -i inventory.ini security_hardening.yml

# Запуск у режимі перевірки (dry-run) — без реальних змін
ansible-playbook -i inventory.ini security_hardening.yml --check --diff

# Запуск лише для конкретної групи серверів
ansible-playbook -i inventory.ini security_hardening.yml --limit webservers

Terraform: інфраструктура з безпековими контролями

Terraform — інструмент від HashiCorp для декларативного опису та автоматичного розгортання хмарної інфраструктури. На відміну від Ansible, який фокусується на конфігурації існуючих серверів, Terraform створює саму інфраструктуру (віртуальні мережі, сервери, бази даних) з вбудованими безпековими контролями.

# main.tf — приклад створення Security Group в AWS з обмеженим доступом

provider "aws" {
  region = "eu-central-1"    # Франкфурт — найближчий регіон до України
}

# Security Group для веб-сервера
resource "aws_security_group" "web_server_sg" {
  name        = "web-server-security-group"
  description = "Security group for web server with restricted access"
  vpc_id      = aws_vpc.main.id

  # Вхідний трафік: дозволити HTTPS з будь-якого джерела
  ingress {
    description = "HTTPS from anywhere"
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  # Вхідний трафік: HTTP перенаправлення (лише для redirect на HTTPS)
  ingress {
    description = "HTTP for redirect only"
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  # Вхідний трафік: SSH лише з IP офісу
  ingress {
    description = "SSH from office IP only"
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["203.0.113.0/24"]   # IP-діапазон офісу
  }

  # Вихідний трафік: дозволити все
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = {
    Name        = "WebServerSG"
    Environment = "Production"
    ManagedBy   = "Terraform"
  }
}

# Security Group для бази даних — доступ лише з веб-серверів
resource "aws_security_group" "database_sg" {
  name        = "database-security-group"
  description = "Security group for database - access only from web servers"
  vpc_id      = aws_vpc.main.id

  ingress {
    description     = "PostgreSQL from web servers only"
    from_port       = 5432
    to_port         = 5432
    protocol        = "tcp"
    security_groups = [aws_security_group.web_server_sg.id]
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = {
    Name        = "DatabaseSG"
    Environment = "Production"
    ManagedBy   = "Terraform"
  }
}

# Шифрування EBS-томів за замовчуванням
resource "aws_ebs_encryption_by_default" "enabled" {
  enabled = true
}
# Робочий процес Terraform
terraform init      # ініціалізація (завантаження провайдерів)
terraform plan      # перегляд запланованих змін (dry-run)
terraform apply     # застосування змін
terraform destroy   # видалення інфраструктури (обережно!)
IaC та безпека Зберігання конфігурацій у Git забезпечує: повний аудит змін (хто, коли, що змінив), можливість код-рев'ю перед застосуванням змін, швидке відновлення після помилкових змін (rollback), автоматичне тестування конфігурацій на відповідність політикам безпеки (policy-as-code, наприклад, Checkov, tfsec).

1.3. Управління оновленнями (Patch Management)

Управління оновленнями (Patch Management) — це систематичний процес ідентифікації, отримання, тестування та встановлення оновлень програмного забезпечення. За статистикою, понад 60% успішних кібератак експлуатують вразливості, для яких вже існують виправлення, що робить patch management одним із найефективніших заходів захисту.

Інструменти управління оновленнями

Інструмент Платформа Можливості
WSUS (Windows Server Update Services) Windows Централізоване управління оновленнями Windows, групові політики, звітність, затвердження оновлень
apt / unattended-upgrades Debian/Ubuntu Автоматичне встановлення security-оновлень, конфігурація через /etc/apt/apt.conf.d/50unattended-upgrades
yum-cron / dnf-automatic RHEL/CentOS/Fedora Автоматичне застосування оновлень безпеки, email-сповіщення
Ansible Кросплатформний Оркестрація оновлень для гетерогенного середовища, rolling updates з перевіркою стану

Цикл Patch Tuesday

Microsoft випускає оновлення безпеки кожного другого вівторка місяця (Patch Tuesday). Рекомендований процес:

  1. Вівторок (день випуску): аналіз бюлетенів безпеки, визначення критичності для інфраструктури організації
  2. Середа — четвер: тестування оновлень у тестовому середовищі
  3. П'ятниця — понеділок: розгортання на некритичних системах
  4. Наступний тиждень: розгортання на виробничих системах за графіком вікон обслуговування
  5. Верифікація: перевірка працездатності систем після оновлення

Пріоритизація оновлень на основі ризику

Пріоритет Критерії Термін встановлення
Екстрений (P0) Активна експлуатація в реальних атаках (0-day), CVSS 9.0+, зовнішні системи 24-48 годин
Критичний (P1) CVSS 7.0+, публічний експлойт, критичні системи 7 днів
Високий (P2) CVSS 4.0-6.9, важливі системи 14-30 днів
Стандартний (P3) CVSS нижче 4.0, внутрішні системи 30-90 днів
Увага Ніколи не встановлюйте оновлення безпосередньо на виробничих системах без попереднього тестування. Навіть security-патчі можуть спричинити несумісність із прикладним ПЗ або порушити роботу сервісів. Завжди використовуйте тестове середовище та плануйте rollback-процедуру.

2. Реагування на інциденти (Incident Response)

Інцидент інформаційної безпеки — це подія або низка подій, що становлять загрозу конфіденційності, цілісності або доступності інформаційних активів та потребують реагування. Незважаючи на найкращі превентивні заходи, інциденти неминучі, тому здатність організації ефективно реагувати на них є критичним фактором мінімізації збитків.

2.1. Фреймворк NIST SP 800-61

NIST SP 800-61 (Computer Security Incident Handling Guide) визначає шість фаз процесу реагування на інциденти:

Фаза 1: Підготовка (Preparation)

Найважливіша фаза, що виконується до виникнення інциденту:

  • Формування команди реагування на інциденти (IRT / CSIRT)
  • Розробка плану реагування на інциденти (Incident Response Plan)
  • Створення playbooks — покрокових інструкцій для типових інцидентів (ransomware, фішинг, витік даних, DDoS)
  • Підготовка інструментарію: forensic workstation, USB з інструментами, засоби для збору доказів
  • Регулярне проведення навчань та табличних вправ (tabletop exercises)
  • Налаштування каналів комунікації (захищений месенджер, альтернативна email-система)

Фаза 2: Виявлення та аналіз (Detection & Analysis)

  • Джерела виявлення: SIEM-алерти, IDS/IPS, EDR, антивірус, повідомлення користувачів, зовнішні джерела (CERT-UA, партнери)
  • Початкова класифікація: підтвердження інциденту (true/false positive)
  • Визначення масштабу: які системи залучені, які дані скомпрометовані
  • Збір та збереження початкових доказів (volatile data — дамп RAM, мережеві з'єднання)
  • Визначення вектора атаки та ідентифікація тактик за MITRE ATT&CK

Фаза 3: Стримування (Containment)

  • Короткострокове стримування: ізоляція скомпрометованих систем від мережі, блокування шкідливих IP/доменів, деактивація скомпрометованих облікових записів
  • Довгострокове стримування: перебудова скомпрометованих систем із чистих образів, застосування тимчасових обхідних заходів (workarounds), посилення моніторингу
  • Рішення про ізоляцію приймається з урахуванням балансу між зупинкою атаки та впливом на бізнес-процеси

Фаза 4: Ліквідація (Eradication)

  • Видалення шкідливого ПЗ з усіх уражених систем
  • Закриття вразливості, через яку було здійснено атаку (патчинг, зміна конфігурації)
  • Скидання скомпрометованих облікових даних (паролі, ключі, сертифікати)
  • Перевірка на наявність інших точок присутності зловмисника (persistence mechanisms)

Фаза 5: Відновлення (Recovery)

  • Поступове повернення систем у виробничий режим
  • Відновлення даних із верифікованих резервних копій
  • Посилений моніторинг відновлених систем протягом визначеного періоду
  • Підтвердження нормальної роботи всіх сервісів

Фаза 6: Засвоєння уроків (Lessons Learned)

  • Проведення post-incident review (PIR) протягом 1-2 тижнів після закриття інциденту
  • Документування: хронологія подій, вжиті заходи, що спрацювало / не спрацювало
  • Визначення кореневих причин (root cause analysis)
  • Оновлення плану реагування, playbooks, правил SIEM на основі отриманого досвіду
  • Формування звіту про інцидент для керівництва та регуляторів (за необхідності)

2.2. Класифікація інцидентів за критичністю

Рівень Назва Опис Приклади Час реагування

2.4. TheHive — платформа управління інцидентами

TheHive — відкрита платформа для управління інцидентами інформаційної безпеки, що забезпечує колаборативне розслідування, трекінг завдань та інтеграцію з іншими інструментами безпеки. TheHive є стандартом де-факто для SOC з відкритим вихідним кодом.

Ключові концепції TheHive

Концепція Опис Приклад
Cases (Справи) Основна одиниця роботи — контейнер для всієї інформації про інцидент: опис, учасники, теги, TLP-маркер Case #2024-0142: "Ransomware infection on file server"
Tasks (Завдання) Конкретні кроки розслідування, призначені відповідальним з термінами виконання "Зібрати дамп пам'яті сервера FS01", "Перевірити логи VPN"
Observables (Спостережувані) Індикатори, пов'язані з інцидентом: IP-адреси, домени, хеші файлів, email-адреси IP: 185.220.101.42, Hash: a1b2c3d4..., Domain: malware-c2.xyz
Alerts (Алерти) Вхідні повідомлення від SIEM, IDS та інших джерел, що можуть бути перетворені на Cases Wazuh alert: "Multiple failed SSH login attempts from 185.220.101.42"

Інтеграція з Cortex

Cortex — компаньйон-платформа TheHive, що надає автоматизований аналіз спостережуваних (observables). Cortex підтримує десятки аналізаторів (analyzers):

  • VirusTotal: перевірка хешів файлів, URL, IP, доменів на шкідливість
  • AbuseIPDB: перевірка репутації IP-адрес
  • Shodan: отримання інформації про відкриті порти та сервіси IP-адреси
  • MISP: пошук збігів з відомими IoC у базах MISP
  • MaxMind GeoIP: геолокація IP-адрес
  • FileInfo: аналіз метаданих файлів

2.5. MISP — платформа обміну індикаторами загроз

MISP (Malware Information Sharing Platform) — відкрита платформа для збору, зберігання, розповсюдження та обміну індикаторами компрометації (IoC) та інформацією про кіберзагрози (Threat Intelligence) між організаціями та спільнотами.

Ключові компоненти MISP

  • Feeds (Фіди): зовнішні джерела даних про загрози, що автоматично імпортуються в MISP:
    • CIRCL OSINT Feed — загальний фід від CIRCL (Computer Incident Response Center Luxembourg)
    • Botvrij.eu — IoC від голландської кіберспільноти
    • Abuse.ch URLhaus — шкідливі URL
    • CERT-UA фіди — індикатори від української CERT
  • Taxonomies (Таксономії): стандартизовані класифікації для уніфікованого маркування подій:
    • TLP (Traffic Light Protocol) — рівні обмеження поширення (RED, AMBER, GREEN, WHITE)
    • MISP Taxonomy — класифікація типів загроз
    • Admiralty Scale — оцінка достовірності джерела та інформації
  • Galaxies (Галактики): розширені контекстні кластери, що пов'язують IoC з групами загроз, інструментами та тактиками:
    • MITRE ATT&CK Galaxy — маппінг подій на тактики та техніки ATT&CK
    • Threat Actor Galaxy — відомі APT-групи та їхня атрибуція
    • Tool Galaxy — інструменти, що використовуються зловмисниками
Практична інтеграція Типовий ланцюг: SIEM (Wazuh) генерує алерт → алерт надходить у TheHive → аналітик створює Case → Cortex автоматично аналізує observables → знайдені IoC публікуються у MISP → MISP розповсюджує IoC партнерам → партнери оновлюють правила блокування.

3. Основи цифрової криміналістики (Digital Forensics)

Цифрова криміналістика (Digital Forensics) — це галузь кібербезпеки, що займається збором, збереженням, аналізом та представленням цифрових доказів. Криміналістичний аналіз є ключовим елементом розслідування інцидентів, оскільки дозволяє відновити хронологію подій, ідентифікувати дії зловмисника та зібрати докази для можливого судового переслідування.

3.1. Принципи збереження доказів

Chain of Custody (Ланцюг зберігання доказів)

Chain of Custody — це задокументований ланцюг відповідальності за докази від моменту їх збору до подання в суді. Кожна передача доказів фіксується із зазначенням:

  • Хто зібрав/отримав/передав докази
  • Коли (точна дата та час)
  • Де зберігалися докази
  • Як забезпечувалась їхня цілісність (хеші, пломби, опечатані контейнери)

Принципи збереження доказів

  • Порядок збору за волатильністю (Order of Volatility): спочатку збирати найбільш мінливі дані — вміст RAM → мережеві з'єднання → запущені процеси → тимчасові файли → журнали → дані дисків
  • Незмінність оригіналу: ніколи не аналізувати оригінальний носій; працювати лише з верифікованими копіями (bit-for-bit images)
  • Верифікація цілісності: обчислювати та фіксувати хеш-суми (SHA-256) на кожному етапі — при збиранні, копіюванні, перед аналізом
  • Документування: фіксувати кожну дію із зазначенням часу, виконавця та результату
Критично важливо Вимкнення скомпрометованого комп'ютера без попереднього збору дампу пам'яті призводить до безповоротної втрати volatile-доказів (запущені процеси, мережеві з'єднання, ключі шифрування в RAM, активність шкідливого ПЗ). Завжди збирайте дамп пам'яті ДО вимкнення системи.

3.2. Volatility — криміналістика оперативної пам'яті

Volatility — провідний фреймворк для криміналістичного аналізу дампів оперативної пам'яті (Memory Forensics) з відкритим вихідним кодом. Volatility підтримує аналіз дампів пам'яті Windows, Linux та macOS.

Отримання дампу пам'яті

# Windows — за допомогою WinPmem
winpmem_mini_x64.exe memory_dump.raw

# Linux — за допомогою LiME (Linux Memory Extractor)
sudo insmod lime-$(uname -r).ko "path=/tmp/memory_dump.lime format=lime"

# Або через /proc/kcore (менш надійний метод)
sudo dd if=/proc/kcore of=/tmp/memory_dump.raw

Аналіз дампу пам'яті за допомогою Volatility 3

# Визначення профілю ОС (Volatility 3 робить це автоматично)
vol -f memory_dump.raw windows.info

# Список запущених процесів
vol -f memory_dump.raw windows.pslist

# Дерево процесів (відображає батьківські/дочірні зв'язки)
vol -f memory_dump.raw windows.pstree

# Приклад виводу pstree (виявлення підозрілого процесу):
# PID   PPID  Name
# 4     0     System
# 380   4     smss.exe
# 528   380   csrss.exe
# 612   528   winlogon.exe
# 660   612   services.exe
# 1284  660   svchost.exe
# 3456  1284  powershell.exe    ← ПІДОЗРІЛО: powershell запущено з svchost
# 3512  3456  cmd.exe           ← ПІДОЗРІЛО: cmd як дочірній процес powershell

# Мережеві з'єднання (аналог netstat)
vol -f memory_dump.raw windows.netscan

# Приклад виводу netscan:
# Proto  LocalAddr       LocalPort  ForeignAddr      ForeignPort  State      PID
# TCPv4  192.168.1.100   49234      185.220.101.42   443          ESTABLISHED 3456
#                                    ↑ Підозріле з'єднання з зовнішнім IP

# Виявлення прихованих/завершених процесів
vol -f memory_dump.raw windows.psscan

# Аналіз DLL, завантажених процесом
vol -f memory_dump.raw windows.dlllist --pid 3456

# Дамп підозрілого процесу для подальшого аналізу
vol -f memory_dump.raw windows.pslist --pid 3456 --dump

# Пошук рядків у пам'яті процесу
vol -f memory_dump.raw windows.strings --pid 3456

# Аналіз реєстру
vol -f memory_dump.raw windows.registry.hivelist

3.3. Autopsy / The Sleuth Kit — криміналістика дисків

The Sleuth Kit (TSK) — набір консольних інструментів для криміналістичного аналізу файлових систем. Autopsy — графічний інтерфейс для TSK, що спрощує аналіз та надає додаткові можливості автоматизації.

Можливості Autopsy

Функція Опис Застосування
Timeline Analysis Побудова часової лінії подій на основі метаданих файлів (створення, модифікація, доступ) та системних артефактів Відновлення хронології дій зловмисника
File Recovery Відновлення видалених файлів за допомогою аналізу нерозподіленого простору (unallocated space) та файлового карвінгу Пошук видалених шкідливих файлів, інструментів зловмисника
Keyword Search Індексований пошук за ключовими словами у всіх файлах, включаючи нерозподілений простір Пошук паролів, URL, email-адрес, специфічних рядків
Hash Lookup Обчислення хешів файлів та порівняння з відомими базами (NSRL — National Software Reference Library) Ідентифікація відомого легітимного та шкідливого ПЗ
Web Artifacts Аналіз артефактів веб-браузерів: історія, кеш, cookie, завантаження Відстеження дій користувача, виявлення фішингових сайтів
Registry Analysis Аналіз реєстру Windows: автозапуск, встановлене ПЗ, підключені пристрої, останні файли Виявлення persistence mechanisms, відстеження USB-пристроїв 
# Створення образу диска (bit-for-bit copy)
sudo dd if=/dev/sda of=/evidence/disk_image.dd bs=4M status=progress

# Обчислення хеш-суми образу для верифікації
sha256sum /evidence/disk_image.dd > /evidence/disk_image.dd.sha256

# Монтування образу для аналізу (read-only!)
sudo mount -o ro,loop /evidence/disk_image.dd /mnt/evidence

# Використання консольних інструментів TSK
# Список розділів
mmls /evidence/disk_image.dd

# Аналіз файлової системи
fls -r -o 2048 /evidence/disk_image.dd

# Створення timeline
fls -r -m "/" -o 2048 /evidence/disk_image.dd > body.txt
mactime -b body.txt -d > timeline.csv

4. Центр операцій безпеки (SOC)

Security Operations Center (SOC) — спеціалізований підрозділ організації, що здійснює безперервний моніторинг, виявлення, аналіз та реагування на інциденти інформаційної безпеки. SOC є «нервовим центром» кібербезпеки організації, що об'єднує людей, процеси та технології.

4.1. Структура SOC: рівні аналітиків

Рівень Роль Основні функції Типові інструменти
L1 — Triage / Alert Monitoring Аналітик першої лінії Моніторинг алертів SIEM у режимі 24/7, початковий тріаж (true/false positive), ескалація підтверджених інцидентів на L2, документування базових подій SIEM (дашборди), тикет-система, runbooks
L2 — Investigation Аналітик-розслідувач Глибокий аналіз ескальованих інцидентів, кореляція подій з різних джерел, визначення масштабу та впливу інциденту, координація стримування SIEM (розширений аналіз), TheHive, EDR, мережевий аналіз
L3 — Threat Hunting / Advanced Analysis Старший аналітик / Threat Hunter Проактивний пошук прихованих загроз (threat hunting), аналіз шкідливого ПЗ, розробка нових правил детектування, участь у розробці стратегії безпеки Volatility, IDA Pro, YARA, Sigma rules, Jupyter notebooks

4.2. Ключові метрики SOC

Метрика Визначення Цільове значення Як вимірюється
MTTD (Mean Time to Detect) Середній час від початку інциденту до його виявлення Менше 24 годин (ідеально — хвилини для критичних) Різниця між часом першого IoC та часом створення алерту
MTTR (Mean Time to Respond) Середній час від виявлення інциденту до початку активного реагування P1: 15 хв, P2: 1 год, P3: 4 год Різниця між часом алерту та часом початку дій з реагування
False Positive Rate Відсоток хибних спрацювань серед усіх алертів Менше 30% (ідеально — менше 10%) Кількість false positive / загальна кількість алертів
MTTC (Mean Time to Contain) Середній час до стримування інциденту P1: 1 год, P2: 4 год Різниця між початком реагування та ізоляцією загрози
Alert Volume Кількість алертів за період Тренд зниження при стабільному покритті Підрахунок у SIEM по періодах

4.3. Робочий процес SOC

Типовий робочий процес обробки алерту в SOC:

  1. Alert (Алерт): SIEM/IDS/EDR генерує алерт на основі кореляційного правила або сигнатури
  2. Triage (Тріаж): L1-аналітик отримує алерт, виконує початковий аналіз за runbook:
    • Це відома хибна тривога? → Закрити як False Positive
    • Це підтверджена подія безпеки? → Створити тикет, ескалювати на L2
    • Недостатньо інформації? → Зібрати додаткові дані, перекваліфікувати
  3. Investigate (Розслідування): L2-аналітик проводить глибокий аналіз:
    • Кореляція з іншими подіями та джерелами
    • Визначення масштабу (які системи залучені)
    • Ідентифікація вектора атаки та TTP (MITRE ATT&CK)
  4. Escalate/Close (Ескалація/Закриття):
    • Підтверджений інцидент → активація Incident Response процесу
    • Не є інцидентом → закриття з документуванням причини
  5. Document (Документування): усі дії, рішення та результати фіксуються у тикет-системі для аудиту та статистики
Приклад роботи SOC: виявлення ransomware
  1. 10:15 — EDR (Wazuh) генерує алерт: "Suspicious file encryption activity detected on host WS-042"
  2. 10:17 — L1 аналітик бачить алерт у дашборді SIEM, відкриває деталі
  3. 10:20 — L1 бачить масове перейменування файлів з розширенням .encrypted, класифікує як P1, ескалює на L2
  4. 10:22 — L2 аналітик ізолює хост WS-042 від мережі через EDR (network containment)
  5. 10:25 — L2 перевіряє інші хости в підмережі — виявлено 3 додаткові заражені системи
  6. 10:30 — IR Manager активує Incident Response Plan для ransomware
  7. 10:35 — Ізоляція всієї підмережі, блокування C2-домену на файрволі
  8. 11:00 — L3 починає аналіз семплу шкідливого ПЗ

5. Модернізація КСЗІ

КСЗІ не є статичною системою — ландшафт загроз постійно змінюється, з'являються нові вразливості та техніки атак, змінюється інфраструктура організації. Тому модернізація КСЗІ є безперервним процесом, що забезпечує адекватність системи захисту актуальним загрозам.

5.1. Періодичний перегляд загроз

  • Щоквартальний аналіз: огляд нових загроз та вразливостей, що стосуються технологій організації; перевірка актуальності моделі загроз
  • Щорічний повний перегляд: оновлення моделі загроз та моделі порушника, переоцінка ризиків, перевірка відповідності нормативним вимогам
  • Позаплановий перегляд: після серйозного інциденту, після значних змін в інфраструктурі, при появі нових нормативних вимог

5.2. Оновлення технологій (Technology Refresh)

  • Заміна засобів захисту, що досягли End-of-Life / End-of-Support
  • Оновлення апаратного забезпечення відповідно до зростаючих навантажень
  • Впровадження нових категорій захисту (наприклад, перехід від традиційного антивірусу до EDR/XDR)
  • Оцінка та впровадження нових технологій: контейнеризація, мікросервіси, хмарні платформи з урахуванням безпеки

5.3. Управління змінами (Change Management)

Будь-які зміни в КСЗІ повинні проходити через формалізований процес управління змінами:

  1. Запит на зміну (RFC): опис зміни, обґрунтування, оцінка впливу на безпеку
  2. Аналіз та затвердження: розгляд комітетом з управління змінами (CAB), оцінка ризиків
  3. Тестування: перевірка зміни у тестовому середовищі
  4. Впровадження: застосування зміни у визначене вікно обслуговування
  5. Верифікація: підтвердження успішності зміни, моніторинг побічних ефектів
  6. Документування: оновлення документації КСЗІ, конфігураційних записів

6. Тренди та перспективи розвитку

Кібербезпека — одна з галузей, що найдинамічніше розвивається. Розглянемо ключові тренди, що впливатимуть на розвиток КСЗІ у найближчі роки.

6.1. XDR (Extended Detection and Response)

XDR — еволюція підходу до виявлення та реагування, що об'єднує дані з множини джерел (endpoints, мережа, хмара, email, identity) у єдину платформу з кореляцією та автоматизованим реагуванням.

Характеристика EDR SIEM XDR
Джерела даних Лише кінцеві точки Логи з різних джерел Кінцеві точки + мережа + хмара + email + identity
Кореляція На рівні endpoint На рівні логів (правила кореляції) Крос-рівнева автоматична кореляція
Реагування Ізоляція хоста, блокування процесів Алерти, інтеграція з SOAR Автоматизоване реагування на всіх рівнях
Приклади CrowdStrike Falcon, Wazuh Splunk, ELK Stack Microsoft Sentinel, Palo Alto Cortex XDR

6.2. AI/ML у кібербезпеці

Штучний інтелект та машинне навчання трансформують як атакуючі, так і захисні технології:

Застосування AI/ML для захисту

  • Поведінковий аналіз (UEBA — User and Entity Behavior Analytics): виявлення аномалій у поведінці користувачів та систем на основі базових моделей (baseline), що дозволяє детектувати інсайдерські загрози та скомпрометовані облікові записи
  • Автоматична класифікація алертів: ML-моделі, що навчені на історичних даних SOC, автоматично класифікують алерти як true/false positive, знижуючи навантаження на аналітиків L1
  • Виявлення шкідливого ПЗ: аналіз поведінки файлів у пісочниці з використанням ML для виявлення zero-day загроз, що не детектуються сигнатурами
  • Threat Intelligence: NLP (Natural Language Processing) для автоматичного аналізу звітів про загрози, виділення IoC, класифікації APT-груп

AI як загроза

  • Deepfake у соціальній інженерії: використання генеративного AI для створення переконливих фішингових повідомлень, голосових та відео-підробок
  • Автоматизація атак: AI-асистенти для генерації експлойтів, обходу засобів захисту, автоматичного pivoting
  • Adversarial ML: атаки на ML-моделі, що використовуються для захисту (evasion, poisoning)

6.3. Платформи Threat Intelligence

Сучасні організації все більше покладаються на структуровану інформацію про кіберзагрози (Cyber Threat Intelligence, CTI) для проактивного захисту:

  • MISP — розглянутий раніше, відкрита платформа обміну IoC
  • OpenCTI — відкрита платформа для управління CTI, інтеграція з MITRE ATT&CK, STIX/TAXII
  • STIX/TAXII — стандарти обміну threat intelligence: STIX (Structured Threat Information eXpression) — формат даних, TAXII (Trusted Automated eXchange of Indicator Information) — протокол обміну
  • Комерційні платформи: Recorded Future, Mandiant Threat Intelligence, CrowdStrike Falcon Intelligence
Контекст України CERT-UA (Урядова команда реагування на комп'ютерні надзвичайні події) є ключовим джерелом threat intelligence для українських організацій. CERT-UA публікує бюлетені про актуальні загрози, IoC від APT-груп, що атакують українську інфраструктуру, та рекомендації щодо захисту. Інтеграція фідів CERT-UA у MISP/SIEM є рекомендованою практикою для всіх організацій.
