Лекція 8. Проєктування КСЗІ: від обстеження до технічного проєкту

Передпроєктне обстеження, технічне завдання та вибір засобів захисту

1. Передпроєктне обстеження ІКС

Проєктування комплексної системи захисту інформації (КСЗІ) починається з передпроєктного обстеження — системного вивчення об'єкта захисту, його інформаційно-комунікаційної системи, інформаційних потоків та існуючих засобів захисту. Результати обстеження є основою для формування вимог до КСЗІ та прийняття архітектурних рішень.

Згідно з нормативними документами системи технічного захисту інформації (НД ТЗІ), передпроєктне обстеження є першим обов'язковим етапом створення КСЗІ і включає:

  1. Інвентаризацію активів
  2. Категоріювання інформації
  3. Аналіз існуючих засобів захисту (gap analysis)
  4. Визначення об'єктів захисту та їх критичності

1.1. Інвентаризація активів

Інвентаризація — це процес виявлення, ідентифікації та каталогізації усіх активів організації, що підлягають захисту. Активи поділяються на три основні категорії:

Апаратні активи

  • Серверне обладнання — фізичні та віртуальні сервери, системи зберігання даних (SAN, NAS), кластерні системи
  • Мережеве обладнання — маршрутизатори, комутатори, міжмережеві екрани, точки доступу Wi-Fi, балансувальники навантаження
  • Кінцеві пристрої — робочі станції, ноутбуки, тонкі клієнти, мобільні пристрої, принтери, МФУ
  • Периферійне обладнання — сканери, USB-пристрої, зовнішні носії
  • Інженерна інфраструктура — ДБЖ, системи кондиціонування, СКУД, системи відеоспостереження

Програмні активи

  • Операційні системи — типи, версії, рівні оновлень на кожному хості
  • Серверне ПЗ — бази даних, веб-сервери, поштові сервери, сервери додатків
  • Прикладне ПЗ — ERP, CRM, документообіг, бухгалтерія, спеціалізовані системи
  • Засоби захисту — антивіруси, EDR, SIEM, DLP, VPN-клієнти
  • Засоби розробки та DevOps — системи CI/CD, контейнери, оркестратори

Інформаційні активи

  • Бази даних — персональні дані, фінансова інформація, медичні записи, комерційна таємниця
  • Документи — внутрішні регламенти, договори, технічна документація
  • Електронне листування — поштові скриньки, архіви листування
  • Резервні копії — місця зберігання, розклад, перевірка відновлення

1.2. Інструменти інвентаризації

Nmap — мережева розвідка та виявлення активів

Nmap (Network Mapper) — найпопулярніший відкритий інструмент для мережевого сканування та виявлення хостів, сервісів та операційних систем. У контексті передпроєктного обстеження Nmap використовується для:

  • Виявлення усіх активних хостів у мережі (host discovery)
  • Визначення відкритих портів та сервісів на кожному хості
  • Ідентифікації операційних систем (OS fingerprinting)
  • Виявлення версій встановленого ПЗ (service version detection)
  • Виявлення незадокументованих або тіньових (shadow IT) пристроїв

Практичні приклади використання Nmap

# 1. Host Discovery — виявлення активних хостів у підмережі
# Ping sweep без сканування портів (-sn)
nmap -sn 192.168.1.0/24

# Результат: список IP-адрес та MAC-адресів активних хостів
# Nmap scan report for 192.168.1.1
#   Host is up (0.0010s latency).
#   MAC Address: AA:BB:CC:DD:EE:01 (Cisco Systems)
# Nmap scan report for 192.168.1.10
#   Host is up (0.0025s latency).
#   MAC Address: AA:BB:CC:DD:EE:10 (Dell)
# 2. Service & OS Detection — визначення сервісів та ОС на конкретному хості
# -sV: визначення версій сервісів
# -O: визначення операційної системи
# -A: aggressive scan (включає -sV, -O, scripts, traceroute)
nmap -sV -O 192.168.1.10

# Результат (приклад):
# PORT     STATE SERVICE       VERSION
# 22/tcp   open  ssh           OpenSSH 8.9p1 Ubuntu 3
# 80/tcp   open  http          Apache httpd 2.4.52
# 443/tcp  open  ssl/http      Apache httpd 2.4.52
# 3306/tcp open  mysql         MySQL 8.0.33
# OS details: Linux 5.15 (Ubuntu 22.04)
# 3. Сканування всієї мережі з визначенням сервісів
# -T4: швидкий темп сканування
# -oA: зберегти результати у всіх форматах (nmap, xml, gnmap)
nmap -sV -T4 -oA network_scan 192.168.0.0/16

# 4. Сканування з використанням NSE-скриптів для виявлення вразливостей
nmap --script vuln 192.168.1.10

# 5. Виявлення SMB-ресурсів (мережеві каталоги Windows)
nmap -p 445 --script smb-enum-shares 192.168.1.0/24
Юридичне застереження Сканування мережі за допомогою Nmap має виконуватися виключно в межах власної інфраструктури або з письмового дозволу власника мережі. Несанкціоноване сканування може бути кваліфіковане як правопорушення відповідно до законодавства України (ст. 361 КК України).

GLPI та Snipe-IT — системи управління ІТ-активами

Після виявлення активів через мережеве сканування, інформація про них має бути каталогізована в системі управління активами:

Система Опис Ключові можливості
GLPI Відкрита система управління ІТ-активами та Service Desk. Має потужний модуль інвентаризації з підтримкою автоматичного виявлення через агенти. Облік обладнання та ПЗ, управління ліцензіями, ITSM (тікет-система), інтеграція з FusionInventory для автоматичної інвентаризації, звітність
Snipe-IT Сучасна відкрита система обліку ІТ-активів з акцентом на простоту та зручність. Написана на Laravel (PHP). Облік обладнання, призначення відповідальних, QR/штрих-коди для фізичної інвентаризації, журнал змін, API, звітність
Практична рекомендація Для повноцінного передпроєктного обстеження рекомендується комбінувати автоматичну інвентаризацію (Nmap + FusionInventory) з ручним оглядом. Автоматичні інструменти можуть не виявити обладнання, що вимкнене, знаходиться в ізольованих сегментах або фільтрує ICMP/сканування.

1.3. Категоріювання інформації

Категоріювання інформації — це процес визначення ступеня обмеження доступу до інформації, що обробляється в ІКС. Від категорії інформації залежить рівень вимог до системи захисту.

Відповідно до законодавства України, інформація за режимом доступу поділяється на:

Категорія Характеристика Приклади Вимоги до захисту
Відкрита інформація Загальнодоступна інформація, обмеження доступу до якої заборонено Публічна звітність, прес-релізи, відкриті реєстри Забезпечення цілісності та доступності
Службова інформація Інформація з обмеженим доступом, що є власністю держави (гриф "Для службового користування") Внутрішні регламенти держорганів, службова переписка Конфіденційність + цілісність + доступність. КСЗІ обов'язкова
Конфіденційна інформація Інформація з обмеженим доступом, що є власністю фізичних/юридичних осіб Персональні дані, комерційна таємниця, банківська таємниця, лікарська таємниця Конфіденційність + цілісність + доступність. КСЗІ обов'язкова
Таємна інформація Державна таємниця — інформація, розголошення якої може завдати шкоди національній безпеці Військові плани, розвідувальні дані, криптографічні ключі Найвищий рівень захисту. Сертифіковані засоби. Спеціальний допуск персоналу
Практичний аспект категоріювання При обстеженні ІКС необхідно визначити, які саме категорії інформації обробляються, де вони зберігаються, якими каналами передаються та хто має до них доступ. Ця інформація безпосередньо впливає на вибір функціонального профілю захищеності та засобів захисту.

1.4. Аналіз існуючих засобів захисту (Gap Analysis)

Gap Analysis (аналіз розривів) — це систематичне порівняння поточного стану захисту ІКС з вимогами, що висуваються до КСЗІ. Мета — виявити "прогалини" (gaps), де існуючий захист не відповідає необхідному рівню.

Процес gap analysis включає:

  1. Документування поточного стану — які засоби захисту вже є, як вони налаштовані, чи оновлюються
  2. Визначення цільового стану — які вимоги висуваються нормативними документами, стандартами, політикою безпеки
  3. Виявлення розривів — де поточний стан не відповідає цільовому
  4. Пріоритизація — ранжування розривів за рівнем ризику
  5. Формування рекомендацій — які засоби захисту необхідно впровадити або модернізувати
Приклад результату gap analysis
Вимога Поточний стан Gap Ризик
Багатофакторна автентифікація Лише паролі MFA відсутня Високий
Шифрування даних at rest Диски не зашифровані Повна відсутність Високий
IDS/IPS Базовий Firewall (iptables) Немає виявлення вторгнень Середній
SIEM Логи зберігаються локально Немає централізованого збору Високий
Резервне копіювання Щотижневе, без перевірки Недостатня частота, немає тестування Середній

1.5. Визначення об'єктів захисту та їх критичності

На основі результатів інвентаризації та категоріювання формується перелік об'єктів захисту — конкретних компонентів ІКС, що потребують захисту. Для кожного об'єкта визначається:

  • Тип активу — сервер, мережевий пристрій, БД, додаток, канал зв'язку
  • Категорія інформації — яка інформація обробляється / зберігається
  • Критичність — наскільки критичним є актив для бізнес-процесів організації
  • Залежності — від яких інших активів залежить та які активи залежать від нього
  • Власник — відповідальна особа / підрозділ

Критичність активу оцінюється за впливом на три властивості інформації (тріада CIA):

Рівень критичності Вплив порушення конфіденційності Вплив порушення цілісності Вплив порушення доступності
Критичний Катастрофічні наслідки: витік держтаємниці, масштабна фінансова втрата Незворотне пошкодження даних, загроза життю Зупинка ключових бізнес-процесів на >24 год
Високий Значний витік: персональні дані, комерційна таємниця Суттєве спотворення даних, що впливає на рішення Зупинка бізнес-процесів на 4-24 год
Середній Витік службової інформації обмеженого масштабу Часткове спотворення, що може бути виправлене Обмеження функціональності на 1-4 год
Низький Витік відкритої або малозначущої інформації Незначне спотворення, легко виправляється Незручності, але без впливу на основні процеси

2. Формування вимог до КСЗІ

2.1. Функціональний профіль захищеності

Відповідно до нормативного документа НД ТЗІ 2.5-004-99 "Критерії оцінки захищеності інформації в комп'ютерних системах від несанкціонованого доступу", вимоги до захисту формулюються через функціональний профіль захищеності — набір функціональних послуг (сервісів) безпеки, що повинна реалізовувати КСЗІ.

Функціональні послуги безпеки згруповані за чотирма напрямками:

Конфіденційність (К)

Позначення Назва Опис
КА Базова адміністративна конфіденційність Адміністратор визначає, хто має доступ до якої інформації. Реалізується через механізми дискреційного управління доступом (DAC).
КО Повторне використання об'єктів Забезпечує неможливість отримання залишкової інформації з об'єктів (очищення пам'яті, диску після видалення).
КВ Довірча конфіденційність Мандатне управління доступом (MAC). Система примусово контролює потоки інформації на основі міток конфіденційності.

Цілісність (Ц)

Позначення Назва Опис
ЦА Базова адміністративна цілісність Адміністратор визначає, хто може модифікувати яку інформацію. Контроль цілісності на рівні DAC.
ЦО Відкат (Rollback) Можливість повернення об'єкта до попереднього стану (версіонування, резервне копіювання).
ЦВ Довірча цілісність Мандатне управління цілісністю. Система примусово контролює модифікацію на основі міток цілісності.

Доступність (Д)

Позначення Назва Опис
ДР Використання ресурсів Контроль за використанням обчислювальних ресурсів (квоти на дисковий простір, обмеження кількості процесів).
ДС Стійкість до відмов Забезпечення працездатності системи при відмові окремих компонентів (резервування, кластеризація).
ДЗ Гаряча заміна Можливість заміни компонентів без зупинки системи (hot swap, hot standby).

Спостереженість (Н)

Позначення Назва Опис
НР Реєстрація Реєстрація подій безпеки: вхід/вихід, доступ до ресурсів, зміна прав, адміністративні дії.
НИ Ідентифікація та автентифікація Однозначна ідентифікація кожного користувача та перевірка його справжності.
НК Керування доступом з КЗЗ Контроль взаємодії з комплексом засобів захисту (КЗЗ): хто і як може змінювати налаштування безпеки.
НО Розподіл обов'язків Розмежування ролей адміністратора безпеки, системного адміністратора та користувачів.
НЦ Цілісність КЗЗ Забезпечення цілісності компонентів самої системи захисту (самозахист КСЗІ).
НТ Самотестування Здатність КСЗІ перевіряти коректність свого функціонування.
НВ Автентифікація під час обміну Взаємна автентифікація при обміні даними між компонентами системи.
Приклад функціонального профілю захищеності Для ІКС, що обробляє конфіденційну інформацію, типовий профіль може виглядати так:

{КА-2, КО-1, ЦА-2, ЦО-1, ДР-1, ДС-1, НР-2, НИ-2, НК-1, НО-1, НЦ-1, НТ-1, НВ-1}

Де число після позначення послуги вказує на рівень реалізації (від 1 — мінімальний до 3 — максимальний). Рівень визначається на основі категорії інформації та результатів аналізу ризиків.

2.2. Рівні гарантій

Окрім функціональних вимог, НД ТЗІ визначає рівні гарантій (Г-1 через Г-7), що описують ступінь впевненості в тому, що КСЗІ коректно реалізує заявлені функції безпеки.

Рівень Характеристика Типове застосування
Г-1 Мінімальний рівень. Функціональне тестування. Системи з відкритою інформацією, низький ризик
Г-2 Структурне тестування. Аналіз архітектури. Системи зі службовою інформацією
Г-3 Методичне тестування. Перевірка механізмів захисту. Системи з конфіденційною інформацією
Г-4 Методичне проєктування. Часткова верифікація. Системи з критично важливою конфіденційною інформацією
Г-5 Напівформальна верифікація. Модульне тестування. Системи з таємною інформацією
Г-6 Формальна верифікація проєкту. Системи з особливо важливою таємною інформацією
Г-7 Формальна верифікація реалізації. Найвищий рівень. Системи спеціального призначення, критична інфраструктура
Зв'язок між категорією інформації та рівнем гарантій Загалом, чим вища категорія інформації — тим вищий рівень гарантій вимагається. Для конфіденційної інформації зазвичай вимагається Г-2 або Г-3. Для державної таємниці — Г-4 та вище. Конкретний рівень визначається за результатами аналізу ризиків та вимог замовника.

3. Технічне завдання на КСЗІ

3.1. Призначення та роль ТЗ

Технічне завдання (ТЗ) — це основний документ, що визначає вимоги до КСЗІ, порядок та умови її розробки. ТЗ є юридичним документом, на основі якого розробляється, впроваджується та приймається КСЗІ. Від якості ТЗ безпосередньо залежить якість створеної системи захисту.

ТЗ на КСЗІ розробляється за результатами передпроєктного обстеження та погоджується між замовником та виконавцем робіт зі створення КСЗІ.

3.2. Структура технічного завдання

Типова структура ТЗ на КСЗІ включає такі розділи:

1. Загальні відомості

  • Повна назва системи та її умовне позначення
  • Найменування замовника та виконавця
  • Перелік документів, на підставі яких створюється КСЗІ
  • Планові строки початку та закінчення робіт
  • Порядок фінансування та оплати робіт

2. Підстава для розробки

  • Посилання на нормативні документи (закони, НД ТЗІ, стандарти)
  • Наказ або рішення про створення КСЗІ
  • Результати передпроєктного обстеження (акт обстеження)

3. Призначення КСЗІ

  • Мета створення — захист інформації з обмеженим доступом в ІКС
  • Функціональне призначення: від яких загроз та порушників має захищати система
  • Об'єкти захисту: перелік компонентів ІКС, інформаційних ресурсів та процесів

4. Вимоги до КСЗІ

  • Функціональний профіль захищеності — перелік та рівні функціональних послуг безпеки
  • Рівень гарантій
  • Вимоги до підсистем: управління доступом, криптографічного захисту, моніторингу, резервування
  • Вимоги до суміщення з існуючою інфраструктурою
  • Вимоги до продуктивності — засоби захисту не повинні деградувати продуктивність ІКС нижче визначеного рівня
  • Вимоги до надійності — час напрацювання на відмову, час відновлення

5. Вимоги до документації

  • Перелік проєктної документації, що має бути розроблена
  • Вимоги до формату, мови, обсягу документів
  • Пояснювальна записка, схеми, специфікації, інструкції

6. Етапи робіт

  • Перелік етапів створення КСЗІ з визначенням змісту робіт на кожному
  • Строки виконання кожного етапу
  • Результати (документи, продукти) кожного етапу

7. Порядок приймання

  • Склад приймальної комісії
  • Програма та методика випробувань
  • Вимоги до оформлення результатів

3.3. Практичні поради щодо складання ТЗ

Поради для якісного ТЗ
  • Конкретність: уникайте розмитих формулювань типу "забезпечити надійний захист". Замість цього: "забезпечити автентифікацію з використанням MFA (TOTP або FIDO2) для всіх користувачів, що мають доступ до конфіденційної інформації".
  • Вимірюваність: кожна вимога повинна мати критерій перевірки. Наприклад: "час виявлення інциденту (MTTD) не повинен перевищувати 15 хвилин".
  • Реалістичність: вимоги мають бути досяжними в межах бюджету та строків. Нереалістичні вимоги призводять до формального "виконання".
  • Повнота: не забувайте про нефункціональні вимоги — продуктивність, масштабованість, сумісність, зручність адміністрування.
  • Трасованість: кожна вимога має бути прив'язана до конкретної загрози з моделі загроз або вимоги нормативного документа.
Типові помилки при розробці ТЗ
  • Копіювання вимог без адаптації до конкретної ІКС
  • Відсутність прив'язки до результатів обстеження та моделі загроз
  • Надмірна деталізація технічних рішень замість формулювання вимог (ТЗ має описувати "що", а не "як")
  • Ігнорування вимог до інтеграції з існуючою інфраструктурою
  • Відсутність критеріїв приймання

4. Технічний проєкт КСЗІ

4.1. Архітектурні рішення

Технічний проєкт — це основний проєктний документ, що визначає архітектуру КСЗІ, обрані засоби захисту та порядок їх розгортання. Архітектура КСЗІ будується на принципах ешелонованого захисту (Defense in Depth).

Принцип Defense in Depth (Ешелонований захист)

Ешелонований захист передбачає створення кількох рівнів (шарів) захисту, кожен з яких незалежно протистоїть загрозам. Якщо зловмисник подолає один рівень, його зупинить наступний.

Рівень захисту Компоненти Призначення
1. Фізичний СКУД, відеоспостереження, замки, охорона Запобігання несанкціонованому фізичному доступу
2. Периметр мережі Firewall (NGFW), WAF, Anti-DDoS, DMZ Фільтрація зовнішнього трафіку, захист від зовнішніх атак
3. Внутрішня мережа Сегментація (VLAN/мікросегментація), IDS/IPS, NAC Контроль внутрішнього трафіку, обмеження lateral movement
4. Хост EDR, антивірус, харденінг ОС, FIM, host-based firewall Захист кінцевих точок та серверів
5. Додаток WAF, API Gateway, автентифікація, авторизація Захист на рівні прикладного ПЗ
6. Дані Шифрування (at rest, in transit), DLP, класифікація, резервне копіювання Захист інформації незалежно від інших рівнів
7. Управління та моніторинг SIEM, SOAR, управління конфігураціями, патч-менеджмент Видимість, контроль та реагування

4.2. Вибір засобів захисту

Вибір конкретних засобів захисту для КСЗІ — це критичний етап, що вимагає системного підходу та врахування множини факторів.

Критерії вибору засобів захисту

Критерій Опис Приклад оцінки
Функціональність Чи покриває засіб усі необхідні функції, визначені в ТЗ Firewall повинен підтримувати L7-фільтрацію, IPS, SSL-інспекцію
Сумісність Інтеграція з існуючою інфраструктурою та іншими засобами захисту SIEM повинен підтримувати збір логів з Windows, Linux, мережевого обладнання
Масштабованість Здатність зростати разом з організацією без заміни рішення EDR-рішення підтримує від 50 до 10000 агентів
Вартість (TCO) Повна вартість володіння: ліцензії + обладнання + впровадження + підтримка + навчання Open-source Wazuh vs. комерційний Splunk — різниця в TCO може бути 10x
Підтримка вендора Якість технічної підтримки, SLA, наявність документації, спільноти Наявність підтримки українською мовою, представництво в Україні
Сертифікація Наявність сертифікатів відповідності (ДССЗЗІ, Common Criteria, FIPS) Для держорганів — наявність експертного висновку ДССЗЗІ є обов'язковою
Зручність адміністрування Складність розгортання, налаштування та повсякденного управління Наявність веб-інтерфейсу, API, централізованого управління
Продуктивність Вплив на продуктивність системи, що захищається EDR не повинен збільшувати час завантаження ОС більш ніж на 10%
Open-Source vs. Комерційні рішення У контексті КСЗІ слід об'єктивно оцінювати як відкриті, так і комерційні рішення. Відкриті рішення (Wazuh, Suricata, pfSense) мають переваги у вартості, прозорості коду та відсутності vendor lock-in. Комерційні рішення (CrowdStrike, Splunk, Palo Alto) пропонують готову підтримку, SLA та інтегровані екосистеми. Оптимальний підхід — комбінувати обидва типи, обираючи найкраще рішення для кожної задачі.

4.3. Типова мережева архітектура із засобами захисту

Нижче описана типова архітектура мережі організації з інтегрованими засобами захисту, що може слугувати основою для технічного проєкту КСЗІ:

Зона 1: Зовнішній периметр (Internet Edge)

  • Border Router — маршрутизація зовнішнього трафіку, базова ACL-фільтрація (anti-spoofing, bogon filtering)
  • NGFW (Next-Generation Firewall) — глибока інспекція трафіку (DPI), IPS, SSL-інспекція, URL-фільтрація, geo-blocking. Приклади: pfSense (open-source), Palo Alto PA-series, Fortinet FortiGate
  • Anti-DDoS — захист від об'ємних атак на рівні провайдера або на периметрі
  • WAF (Web Application Firewall) — захист веб-додатків від SQL-injection, XSS, CSRF. Приклади: ModSecurity, Cloudflare WAF

Зона 2: DMZ (Demilitarized Zone)

  • Публічно доступні сервіси: веб-сервер, поштовий шлюз, DNS, reverse proxy
  • Ізольована від внутрішньої мережі окремими правилами firewall
  • Трафік з DMZ до внутрішньої мережі — мінімально необхідний та суворо контрольований

Зона 3: Внутрішня мережа (Internal Network)

  • Core Switch з підтримкою VLAN для сегментації
  • Сегменти:
    • User VLAN — робочі станції користувачів
    • Server VLAN — внутрішні сервери (AD, файловий сервер, БД, ERP)
    • Management VLAN — інтерфейси управління обладнанням (out-of-band management)
    • Security VLAN — SIEM, SOAR, сервери безпеки
    • VoIP VLAN — IP-телефонія (якщо є)
  • IDS/IPS (Suricata) — на span-порті core switch для моніторингу east-west трафіку
  • NAC (Network Access Control) — перевірка стану пристроїв перед допуском до мережі

Зона 4: Віддалений доступ

  • VPN-шлюз (WireGuard або OpenVPN) — для віддалених працівників
  • Або ZTNA-рішення — для організацій, що впроваджують Zero Trust
  • MFA — обов'язкова для всіх віддалених підключень

Зона 5: Моніторинг та управління

  • SIEM (Wazuh або ELK Stack) — централізований збір та аналіз подій
  • EDR (Wazuh Agent) — на кожній кінцевій точці та сервері
  • Сервер управління конфігураціями (Ansible) — централізоване управління
  • Сервер моніторингу (Zabbix або Prometheus + Grafana) — моніторинг доступності та продуктивності

4.4. Документація технічного проєкту

Технічний проєкт КСЗІ оформлюється у вигляді комплекту документів, що включає:

Документ Зміст
Пояснювальна записка Загальний опис КСЗІ, обґрунтування прийнятих рішень, відповідність вимогам ТЗ, опис архітектури та принципів роботи
Структурна схема КСЗІ Загальна архітектура системи захисту, взаємозв'язки між компонентами
Схема мережевої топології Фізична та логічна топологія мережі із зазначенням розміщення засобів захисту
Функціональна схема Опис функцій кожного компонента КСЗІ та їхньої взаємодії
Специфікація обладнання Перелік апаратних засобів з характеристиками: модель, продуктивність, кількість, вартість
Специфікація програмного забезпечення Перелік ПЗ з версіями, типами ліцензій, кількістю, вартістю
Матриця відповідності Таблиця, що показує, як кожна вимога ТЗ реалізується конкретними засобами захисту
План розгортання Послідовність впровадження, залежності, часові рамки

6. Підсумки та ключові тези

  • Передпроєктне обстеження — фундамент КСЗІ. Без повного розуміння об'єкта захисту неможливо створити ефективну систему. Ключові етапи: інвентаризація, категоріювання, gap analysis.
  • Інструменти інвентаризації (Nmap, GLPI, Snipe-IT) дозволяють автоматизувати процес виявлення та обліку активів.
  • Функціональний профіль захищеності (НД ТЗІ 2.5-004-99) визначає конкретні послуги безпеки, які повинна реалізовувати КСЗІ, за чотирма напрямками: конфіденційність, цілісність, доступність, спостереженість.
  • Технічне завдання повинно бути конкретним, вимірюваним та прив'язаним до результатів обстеження та моделі загроз.
  • Технічний проєкт будується на принципі ешелонованого захисту (Defense in Depth) та включає обґрунтований вибір засобів захисту для кожного рівня.
  • Документація є невід'ємною частиною КСЗІ: пояснювальна записка, схеми, специфікації, матриця відповідності — без них КСЗІ не може бути прийнята в експлуатацію.

  1. Що включає передпроєктне обстеження ІКС? Назвіть та опишіть основні етапи.
  2. Які три категорії активів підлягають інвентаризації при обстеженні ІКС?
  3. Як використовується Nmap при передпроєктному обстеженні? Наведіть приклади команд для host discovery та service detection.
  4. Поясніть відмінності між категоріями інформації: відкрита, службова, конфіденційна, таємна. Як вони впливають на вимоги до КСЗІ?
  5. Що таке gap analysis? Опишіть процес та наведіть приклад.
  6. Поясніть функціональні послуги безпеки за НД ТЗІ 2.5-004-99. Наведіть приклади для кожного з чотирьох напрямків (К, Ц, Д, Н).
  7. Що таке рівні гарантій (Г-1 через Г-7)? Як визначається необхідний рівень?
  8. Опишіть типову структуру технічного завдання на КСЗІ. Які розділи є обов'язковими?
  9. Назвіть типові помилки при розробці ТЗ на КСЗІ та способи їх уникнення.
  10. Поясніть принцип Defense in Depth. Назвіть рівні захисту та засоби для кожного рівня.
  11. За якими критеріями обирають засоби захисту для КСЗІ? Назвіть щонайменше шість критеріїв.
  12. Що таке матриця відповідності? Наведіть приклад (вимога ТЗ, послуга, засіб реалізації).
  13. Опишіть типову мережеву архітектуру організації з інтегрованими засобами захисту. Які зони безпеки виділяються?
