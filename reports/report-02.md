## 2. Політика безпеки

### ПОЛІТИКА ІНФОРМАЦІЙНОЇ БЕЗПЕКИ ТОВ «TechSoft»

Версія 1.0 | Дата: 18.02.2026 | Затвердив: Директор Коваленко О.І.

### Розділ 1. Загальні положення

Ця Політика визначає правила захисту інформаційних ресурсів ТОВ «TechSoft». Мета -- забезпечення конфіденційності, цілісності та доступності інформації. Поширюється на всіх працівників, підрядників та всі ІТ-системи компанії. Базові принципи: мінімальні привілеї, ешелонований захист, розмежування обов'язків.

### Розділ 2. Правила доступу

Доступ надається на підставі заявки, узгодженої керівником відділу. Модель RBAC: доступ через призначення ролі. Перегляд прав -- щоквартально. При звільненні -- деактивація протягом 2 годин. Облікові записи без активності 60+ днів деактивуються.

### Розділ 3. Парольна політика

Мінімум 12 символів, великі/малі літери + цифри + спецсимволи. Зміна кожні 90 днів. Блокування після 5 невдалих спроб на 30 хв. Заборона повтору останніх 10 паролів. 2FA обов'язкова для VPN та хмарних сервісів.

### Розділ 4. Реагування на інциденти

Інцидент ІБ: фішинг, підозріла активність, втрата пристрою, спроба несанкціонованого доступу. Повідомлення протягом 1 години на security@techsoft.ua. Критичні інциденти (витік, ransomware) -- негайна реакція; інші -- до 4 годин.

## 4. Матриця доступу

Позначення: **R** -- читання, **W** -- запис, **Повний** -- повне адміністрування, **--** -- немає доступу.

| Ресурс | Адміністратор | Розробник | Менеджер | Бухгалтер | Директор |
|--------|:---:|:---:|:---:|:---:|:---:|
| Вихідний код (GitHub) | Повний | R/W | R | -- | -- |
| БД клієнтів (AWS RDS) | Повний | R (логи) | -- | -- | -- |
| Бухгалтерська система (1С) | Повний | -- | -- | R/W | R |
| Google Workspace (пошта) | Повний | R/W | R/W | R/W | R/W |
| Файловий сервер | Повний | R/W | R/W | R | R/W |
| Active Directory | Повний | -- | -- | -- | -- |
| VPN-шлюз | Повний | Підключення | Підключення | Підключення | Підключення |

**Примітки:** Розробники мають лише читання логів БД (для діагностики), без доступу до персональних даних. Бухгалтери ізольовані від технічних систем. Директор може переглядати фінансові звіти 1С, але не змінювати дані.

## 5. Правила розмежування обов'язків (SoD)

| № | Функція 1 | Функція 2 | Тип | Обґрунтування |
|---|-----------|-----------|-----|---------------|
| 1 | Написання коду (Розробник) | Розгортання у продакшн (Адміністратор) | Статичний | Запобігання впровадженню шкідливого коду без code review. Розробник не може самостійно розгортати код у продакшн. |
| 2 | Створення платежу (Бухгалтер) | Затвердження платежу (Директор) | Статичний | Запобігання фінансовому шахрайству. Одна особа не може створити і затвердити фінансову операцію. |
| 3 | Управління акаунтами (Адміністратор) | Аудит дій користувачів (Менеджер з ІБ) | Статичний | Незалежний контроль. Адміністратор не повинен мати змогу видалити логи своїх дій. |

## 6. Відповіді на контрольні питання

### 1. У чому різниця між DAC, MAC та RBAC?

**DAC (Discretionary Access Control)** -- власник ресурсу самостійно визначає права доступу (приклад: chmod у Linux, NTFS ACL у Windows). Гнучка, але ризикована: власник може надати зайві права.

**MAC (Mandatory Access Control)** -- система приймає рішення на основі рівнів секретності (мітки безпеки). Користувач не може змінити рівень. Приклад: SELinux, військові системи з грифами «Таємно».

**RBAC (Role-Based Access Control)** -- права призначаються ролям, а не окремим користувачам. Користувач отримує доступ через призначення ролі.

Для корпоративного середовища найбільш доцільна RBAC, оскільки вона відповідає організаційній структурі, спрощує адміністрування при великій кількості працівників та забезпечує принцип мінімальних привілеїв.

### 2. Що таке принцип мінімальних привілеїв?

Принцип мінімальних привілеїв (Least Privilege) означає, що кожен суб'єкт повинен мати лише ті права, які мінімально необхідні для виконання його функцій. У матриці доступу це реалізується так: розробник має R/W лише до коду та тестових середовищ, але не має доступу до 1С чи AD; бухгалтер працює лише з 1С та поштою, без доступу до GitHub або AWS. Це обмежує потенційну шкоду при компрометації будь-якого облікового запису.

### 3. Що таке SoD?

Розмежування обов'язків (Separation of Duties) -- принцип, за яким критичні операції розподіляються між кількома особами. **Статичний SoD**: заборона суміщення ролей -- наприклад, розробник не може одночасно бути адміністратором продакшн. **Динамічний SoD**: користувач може мати обидві ролі, але не може активувати їх одночасно -- наприклад, QA-інженер може також мати роль менеджера проєкту, але не може одночасно тестувати продукт і приймати рішення про його випуск.

### 4. Які розділи повинна містити політика безпеки?

Основні розділи: загальні положення (мета, принципи), сфера дії, ролі та відповідальність, правила управління доступом, парольна політика, правила допустимого використання, реагування на інциденти, санкції. Політику має затверджувати керівник організації, оскільки: це надає документу юридичну силу для всіх працівників; демонструє підтримку керівництва (вимога ISO 27001); ІТ-відділ не має повноважень встановлювати обов'язкові правила для всієї організації.

### 5. Як тріада CIA допомагає визначити вимоги до захисту?

Тріада CIA дозволяє системно оцінити потреби захисту кожного активу за трьома незалежними критеріями. Наприклад, для Active Directory всі три вимоги критичні (компрометація дає повний доступ). Для файлового сервера конфіденційність середня (загальні документи), але цілісність висока (зміна шаблонів порушує процеси). Такий підхід допомагає розставити пріоритети: актив з критичною оцінкою за всіма трьома критеріями потребує максимального захисту, тоді як актив із середніми оцінками може мати менш суворі заходи.

## Висновки

У ході виконання практичної роботи було розроблено комплекс документів з інформаційної безпеки для ТОВ «TechSoft»:

1. **Визначено 7 інформаційних активів** з оцінкою вимог CIA. Найбільш критичні: Active Directory (К/Ц/Д = Критична), база даних клієнтів та бухгалтерська система.

2. **Написано політику безпеки** з 4 розділів: загальні положення, правила доступу, парольна політика, реагування на інциденти.

3. **Визначено 5 ролей** (Адміністратор, Розробник, Менеджер, Бухгалтер, Директор) та побудовано **матрицю доступу** 7 ресурсів x 5 ролей з дотриманням принципу мінімальних привілеїв.

4. **Сформульовано 3 правила SoD**, що запобігають концентрації критичних повноважень: розділення розробки та розгортання, розділення створення та затвердження платежів, розділення адміністрування та аудиту.

Рольова модель RBAC дозволяє ефективно управляти доступом для 30 працівників, забезпечуючи баланс між безпекою та зручністю використання ІКС.
