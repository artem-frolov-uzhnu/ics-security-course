Вибір та обґрунтування засобів захисту для організації

Мета роботи

Навчитися обирати та обґрунтовувати засоби захисту для конкретної організації. Студенти мають описати об'єкт захисту, визначити ключові ІТ-системи та загрози, обрати 5 засобів захисту з різних категорій, побудувати схему мережі з позначенням розміщення засобів та скласти стислий план впровадження.

Теоретичні відомості

Комплексна система захисту інформації (КСЗІ)

КСЗІ — це сукупність організаційних, інженерно-технічних та програмних заходів, спрямованих на забезпечення захисту інформації в інформаційно-комунікаційній системі. Проєктування КСЗІ базується на результатах аналізу загроз, визначення вимог до захисту та оцінки ризиків.

Згідно з НД ТЗІ 3.7-003-2005, процес створення КСЗІ включає чотири основні стадії:

  1. Передпроєктна стадія — обстеження об'єкта, аналіз інфраструктури, формування вимог
  2. Стадія проєктування — вибір засобів захисту, розробка архітектури
  3. Стадія впровадження — монтаж, налаштування, випробування
  4. Стадія супроводу — експлуатація, моніторинг, модернізація

Принцип ешелонованого захисту (Defense in Depth)

Ключовий принцип побудови КСЗІ — ешелонований (багатошаровий) захист. Ідея полягає в тому, що кожен рівень захисту є незалежним бар'єром: якщо зловмисник подолає один рівень, наступний продовжує забезпечувати безпеку. Типові рівні:

  • Периметр мережі — міжмережевий екран, IDS/IPS
  • Мережевий рівень — VPN, сегментація мережі (VLAN)
  • Рівень кінцевих точок — антивірус, EDR, повнодискове шифрування
  • Рівень даних — резервне копіювання, шифрування, DLP
  • Організаційний рівень — політики, навчання персоналу, реагування на інциденти
Чому це важливо Жоден окремий засіб захисту не може гарантувати повну безпеку. Наприклад, міжмережевий екран не захистить від інсайдерської загрози, а антивірус не зупинить DDoS-атаку. Тому КСЗІ завжди включає засоби різних категорій, які доповнюють одне одного.

Основні категорії засобів захисту

При проєктуванні КСЗІ необхідно обрати засоби захисту з кожної основної категорії:

Категорія Призначення Приклади
Міжмережевий екран (Firewall) Контроль та фільтрація мережевого трафіку між зонами pfSense, OPNsense, Fortinet FortiGate, Cisco ASA
IDS/IPS Виявлення та запобігання вторгненням на основі сигнатур та аномалій Suricata, Snort, Zeek
VPN Шифрування каналів зв'язку для віддаленого доступу WireGuard, OpenVPN, IPsec
Антивірус / EDR Захист кінцевих точок від шкідливого ПЗ, виявлення підозрілої поведінки CrowdStrike Falcon, Microsoft Defender for Endpoint, Wazuh Agent
Резервне копіювання (Backup) Збереження копій даних для відновлення після інцидентів Veeam, Restic, Bacula, AWS Backup
Порада При виборі конкретного продукту для кожної категорії враховуйте: розмір організації, бюджет (open-source чи комерційне рішення), сумісність з існуючою інфраструктурою, наявність підтримки та документації.

3. Схема мережі

Текстовий опис схеми мережі (для побудови в draw.io)

1. Інтернет → з'єднується з pfSense Firewall (на апаратній платформі Protectli Vault). pfSense виконує функції stateful firewall, NAT та VPN-шлюзу. Suricata IDS/IPS працює як вбудований модуль pfSense для інспекції вхідного та вихідного трафіку.

2. DMZ (192.168.10.0/24) — окремий порт pfSense. Тут розташовані сервери з доступом ззовні: веб-сервер для клієнтських демо-проєктів. Правила firewall дозволяють трафік з Інтернету лише на порт 443 (HTTPS). DMZ не має прямого доступу до LAN.

3. LAN — офіс (192.168.1.0/24) — 35 робочих станцій, Wi-Fi AP, принтер. На кожній робочій станції встановлено CrowdStrike Falcon Agent (EDR). Доступ до Інтернету через pfSense з фільтрацією.

4. Серверний сегмент (192.168.2.0/24) — VLAN, відокремлений від LAN. Контролер домену Active Directory, файловий сервер, NAS Synology з Veeam Backup. Доступ лише для ІТ-адміністраторів.

5. VPN (WireGuard) — WireGuard шлюз на pfSense. 15 віддалених працівників підключаються через зашифрований тунель та отримують доступ до LAN та серверного сегмента відповідно до своєї ролі.

6. Хмарний сегмент (AWS) — продакшн-середовище клієнтських проєктів (EC2, RDS, S3). З'єднання з офісом через Site-to-Site VPN (WireGuard). Offsite-бекапи зберігаються в AWS S3 Glacier.
Де розміщено кожен засіб
  • pfSense (Firewall) — на периметрі, між Інтернетом та всіма внутрішніми сегментами
  • Suricata (IDS/IPS) — інлайн на pfSense, інспектує трафік WAN → DMZ та WAN → LAN
  • WireGuard (VPN) — шлюз на pfSense для віддалених працівників та site-to-site з AWS
  • CrowdStrike Falcon (EDR) — агент на кожній робочій станції (50 шт.) та серверах
  • Veeam Backup (Backup) — на NAS Synology у серверному сегменті + репліка в AWS S3

4. План впровадження

Етап Назва Засоби захисту Термін Критерій завершення
1 Захист периметра мережі pfSense (firewall), Suricata (IDS/IPS) 2 тижні pfSense налаштовано: VLAN для LAN, DMZ та серверного сегмента; правила фільтрації активні; Suricata працює інлайн з набором правил ET Open; тестовий трафік коректно фільтрується та логується
2 Захист доступу та кінцевих точок WireGuard (VPN), CrowdStrike Falcon (EDR) 2 тижні WireGuard: конфігурації для 15 віддалених працівників згенеровані та протестовані; Site-to-Site VPN з AWS працює. CrowdStrike: агент встановлено на 50 робочих станціях та 2 серверах; консоль показує всі пристрої як захищені
3 Резервне копіювання та перевірка Veeam Backup 1-2 тижні Veeam: щоденне резервне копіювання Active Directory, файлового сервера та конфігурацій на NAS; репліка бекапів у AWS S3 Glacier; тестове відновлення одного сервера з бекапу виконано успішно (RTO ≤ 4 год)

Загальна тривалість впровадження: 5-6 тижнів.

Обґрунтування послідовності етапів

Етап 1 (Периметр) впроваджується першим, оскільки без firewall та IDS/IPS мережа залишається відкритою для будь-яких зовнішніх атак. Це фундамент, на якому будується решта захисту.

Етап 2 (Доступ і кінцеві точки) іде другим: VPN забезпечує безпечний канал для віддалених працівників (без нього вони підключаються через незахищені мережі), а EDR захищає робочі станції від ransomware — найкритичнішої загрози.

Етап 3 (Backup) завершує впровадження: резервне копіювання є «останньою лінією оборони» та гарантує відновлення навіть після успішної атаки.

Вимоги до звіту

Звіт має містити такі розділи (загальний обсяг — 3-5 сторінок):

  1. Титульна сторінка — назва дисципліни, номер та назва практичної роботи, ПІБ студента, група, дата
  2. Мета роботи
  3. Опис організації та ІТ-систем — стислий опис (3-5 речень) + таблиця ключових ІТ-систем (5-7 записів)
  4. Топ-3 загрози — таблиця з назвою, джерелом та впливом
  5. Таблиця обраних засобів захисту — 5 засобів з обґрунтуваннями
  6. Схема мережі — скріншот з draw.io з позначенням розміщення всіх засобів захисту
  7. План впровадження — таблиця з 3 етапами
  8. Відповіді на контрольні питання
  9. Висновки
Важливо Схема мережі має бути побудована в draw.io та вставлена у звіт як зображення. Схеми, намальовані від руки або у текстовому форматі, не приймаються. Збережіть файл draw.io разом зі звітом.

Контрольні питання

  1. Поясніть принцип ешелонованого захисту (Defense in Depth). Як цей принцип реалізовано у вашому проєкті?
  2. За якими критеріями слід обирати засоби захисту для КСЗІ? Порівняйте переваги та недоліки open-source та комерційних рішень.
  3. Чому важливо сегментувати мережу на зони (LAN, DMZ, серверний сегмент)? Як це обмежує поширення атаки?
  4. Що буде, якщо один з обраних засобів захисту вийде з ладу? Які компенсаційні механізми передбачені у вашому проєкті?
  5. Яка роль резервного копіювання у КСЗІ? Поясніть стратегію 3-2-1 та чому offsite-копія є критично важливою.
