Практична робота 1. Моделювання загроз для ІКС

Побудова DFD, ідентифікація загроз за STRIDE

Мета роботи

Навчитися виявляти загрози інформаційно-комунікаційної системи (ІКС) за допомогою методології STRIDE: будувати спрощену діаграму потоків даних (DFD Level 0), ідентифікувати загрози за шістьма категоріями STRIDE, описувати модель порушника та пропонувати контрзаходи для найсерйозніших загроз.

Теоретичні відомості

Діаграма потоків даних (DFD)

DFD (Data Flow Diagram) — це графічне представлення руху даних між компонентами системи. DFD використовується як основа для моделювання загроз, оскільки наочно показує, де дані створюються, обробляються, зберігаються та передаються. У спрощеному вигляді достатньо побудувати діаграму рівня 0 (контекстну), яка показує систему як один процес з усіма зовнішніми сутностями.

Основні елементи DFD:

  • Процес (Process) — компонент, що обробляє дані (зображується колом або прямокутником із заокругленими кутами)
  • Сховище даних (Data Store) — місце зберігання даних: база даних, файлова система (зображується двома горизонтальними лініями)
  • Потік даних (Data Flow) — переміщення даних між елементами (зображується стрілкою з підписом)
  • Зовнішня сутність (External Entity) — джерело або одержувач даних за межами системи: користувач, зовнішній сервіс (зображується прямокутником)
  • Межа довіри (Trust Boundary) — пунктирна лінія, що розділяє зони з різними рівнями довіри
DFD Level 0 (контекстна діаграма) На цьому рівні вся система зображується як один процес. Навколо нього розміщуються зовнішні сутності (користувачі, зовнішні системи), з'єднані потоками даних. Межі довіри показують, де змінюється рівень безпеки — саме на цих межах найчастіше виникають загрози.

Методологія STRIDE

STRIDE — це мнемонічна абревіатура, запропонована Microsoft, для класифікації загроз інформаційній безпеці. Кожна літера відповідає одній категорії загроз:

Категорія Опис Порушена властивість
Spoofing (Підміна) Видавання себе за іншу сутність (користувача, систему) Автентичність
Tampering (Підробка) Несанкціонована зміна даних Цілісність
Repudiation (Відмова від авторства) Заперечення виконаних дій Неспростовність
Information Disclosure (Витік інформації) Несанкціонований доступ до даних Конфіденційність
Denial of Service (Відмова в обслуговуванні) Порушення доступності сервісу Доступність
Elevation of Privilege (Підвищення привілеїв) Отримання несанкціонованих повноважень Авторизація

Оцінка серйозності загроз

У спрощеному варіанті для кожної загрози визначається якісна оцінка серйозності за трьома рівнями:

Рівень серйозності Опис Приклад
Висока Загроза може призвести до значних фінансових збитків, витоку великих масивів даних або повної зупинки системи Ransomware на файловому сервері, витік бази клієнтів
Середня Загроза може спричинити часткові збитки, порушення роботи окремих сервісів або обмежений витік даних DDoS-атака на веб-сервер, несанкціонований доступ до окремих документів
Низька Загроза має обмежений вплив, може бути швидко усунена або стосується некритичних даних Перевантаження VPN-шлюзу, випадковий витік нечутливої інформації

Необхідне ПЗ та інструменти

  • draw.io (diagrams.net) — безкоштовний онлайн-інструмент для побудови діаграм, не потребує реєстрації та встановлення. Використовується для побудови DFD
  • Google Таблиці або Microsoft Excel — для заповнення таблиці загроз та оформлення результатів
Порада У draw.io для DFD використовуйте фігури із розділу General або Flowchart: прямокутники для зовнішніх сутностей, кола або прямокутники з заокругленими кутами для процесів, пунктирні лінії для меж довіри. Підписуйте кожен потік даних.

Варіанти завдань

Студент обирає один із варіантів або отримує варіант від викладача. Обраний об'єкт буде використовуватися також у практичних роботах 2 та 8. Порівняно з повною версією роботи, обсяг спрощено: одна DFD-діаграма замість двох, 8–10 загроз замість 15–20, 2 типи порушників замість 4.

Хід роботи

Крок 1. Опис організації

Оберіть варіант та опишіть організацію у 5–7 реченнях. Опис має включати:

  • Назву організації та сферу діяльності
  • Кількість працівників та географічне розташування
  • Типи інформації, що обробляється (персональні дані, фінансова інформація, комерційна таємниця тощо)
  • Основні ІТ-системи та сервіси (сервери, хмарні платформи, корпоративна пошта)
Порада Не намагайтеся описати все — зосередьтеся на 3–4 ключових системах, які визначають специфіку вашої організації. Наприклад, для IT-компанії це Git-репозиторій та CI/CD, для медичного закладу — медична інформаційна система та IoT-обладнання.

Крок 2. Побудова DFD Level 0

Створіть контекстну діаграму потоків даних у draw.io. Діаграма має містити:

  • 1 центральний процес — ІКС вашої організації
  • 3–5 зовнішніх сутностей — користувачі та зовнішні системи, що взаємодіють з ІКС
  • 1–2 сховища даних — основні місця зберігання інформації
  • Потоки даних — стрілки між елементами з підписами (які саме дані передаються)
  • Межі довіри — пунктирні лінії, що розділяють зони з різним рівнем безпеки (наприклад, Інтернет | Корпоративна мережа)
Увага Кожний потік даних повинен мати підпис, що описує, які саме дані передаються (наприклад: «облікові дані», «HTTP-запити», «вихідний код»). Потоки без підписів — типова помилка.

Крок 3. Ідентифікація загроз за STRIDE

Для кожної категорії STRIDE (6 категорій) ідентифікуйте по 1–2 загрози. Загальна кількість: 8–10 загроз. Заповніть таблицю:

№ Елемент DFD Категорія STRIDE Опис загрози Серйозність
1 Назва елемента Категорія Конкретний опис загрози Висока / Середня / Низька
... ... ... ... ...
Як ідентифікувати загрози Для кожного елемента DFD подумайте: хто може підмінити цю сутність (Spoofing)? Хто може змінити дані (Tampering)? Чи можна заперечити дію (Repudiation)? Хто може отримати несанкціонований доступ до даних (Information Disclosure)? Чи можна зробити сервіс недоступним (Denial of Service)? Чи можна отримати більше прав (Elevation of Privilege)?

Крок 4. Модель порушника

Опишіть два типи потенційних порушників для вашої системи: зовнішнього та внутрішнього. Для кожного типу визначте 3–4 характеристики:

  • Мотивація — чому порушник може атакувати систему
  • Рівень знань та можливостей — які технічні засоби може використовувати
  • Початковий доступ — звідки починається атака
  • Типові сценарії атак — які дії найбільш вірогідні
Приклад визначення типів порушників

Зовнішній порушник: хакер або кіберзлочинна група, мотивована фінансовою вигодою. Має доступ лише з Інтернету до публічних сервісів (веб-сайт, VPN-шлюз, пошта). Використовує фішинг, експлуатацію вразливостей, DDoS.

Внутрішній порушник: незадоволений працівник або жертва соціальної інженерії. Має обліковий запис у корпоративній мережі, доступ до робочої станції та внутрішніх ресурсів. Може скопіювати конфіденційні дані або випадково запустити шкідливе ПЗ.

Крок 5. Контрзаходи для топ-3 загроз

Оберіть 3 найсерйозніші загрози з таблиці (крок 3) та для кожної запропонуйте по 2 контрзаходи — один організаційний та один технічний:

  • Організаційний контрзахід: політика, процедура, навчання персоналу, регламент
  • Технічний контрзахід: конкретний засіб захисту, налаштування, інструмент

Для кожного контрзаходу коротко поясніть, як саме він знижує ризик.

Крок 6. Висновок

Зробіть загальний висновок по роботі:

  • Які категорії STRIDE є найнебезпечнішими для обраної організації і чому?
  • Який тип порушника (зовнішній чи внутрішній) становить більшу загрозу?
  • Які загальні рекомендації щодо захисту ІКС можна надати?

Приклад виконання

Нижче наведено повний приклад виконання для Варіанту 1 — IT-компанія «TechSoft» (~30 працівників).

Зверніть увагу Цей приклад є орієнтовним. У вашій роботі опис, загрози та оцінки мають відповідати саме вашому варіанту завдання. Не копіюйте приклад дослівно — адаптуйте підхід до свого об'єкта захисту.

1. Опис організації

Опис об'єкта захисту

ТОВ «TechSoft» — невелика IT-компанія, що займається розробкою програмного забезпечення на замовлення (веб-додатки та хмарні рішення). У компанії працює близько 30 осіб: 18 розробників, 3 DevOps-інженери, 3 менеджери проєктів, 2 QA-інженери, 2 бухгалтери та 2 ІТ-адміністратори. Центральний офіс розташований у Києві, 10 працівників працюють віддалено з різних міст України. Компанія обробляє комерційну таємницю (вихідний код продуктів), персональні дані працівників та клієнтів, а також фінансову документацію. Основні ІТ-системи: корпоративна пошта Google Workspace, Git-репозиторій на GitHub, хмарна інфраструктура AWS (веб-сервери, бази даних PostgreSQL), файловий сервер та VPN на базі Mikrotik для віддалених працівників.

2. DFD Level 0 — Контекстна діаграма

На рівні 0 вся ІКС компанії «TechSoft» представлена як один процес, що взаємодіє із зовнішніми сутностями.

Текстовий опис DFD Level 0

Центральний процес: [P0] ІКС «TechSoft»

Зовнішні сутності:
[E1] Працівник (офіс) — розробники, менеджери, бухгалтери, що працюють в офісі
[E2] Працівник (віддалений) — розробники та інші працівники на дистанційній роботі
[E3] Клієнт — замовники, що взаємодіють з веб-додатками компанії
[E4] GitHub — зовнішній хмарний сервіс для зберігання коду та CI/CD
[E5] AWS — хмарна платформа для розгортання додатків та зберігання даних

Сховища даних:
[D1] Файловий сервер — внутрішні документи, договори, фінансова звітність
[D2] БД PostgreSQL (AWS RDS) — дані клієнтських веб-додатків

Потоки даних:
[E1] → [P0]: облікові дані, код (git push), документи, фінансові дані
[P0] → [E1]: доступ до ресурсів, результати збірки, звіти
[E2] → [P0]: облікові дані + VPN-тунель, код через VPN
[P0] → [E2]: доступ до ресурсів через VPN
[E3] → [P0]: HTTP/HTTPS-запити, дані форм
[P0] → [E3]: веб-контент, дані API
[P0] ↔ [E4]: git push/pull, webhooks, CI/CD-артефакти
[P0] ↔ [E5]: запити до БД, розгортання додатків, бекапи
[P0] ↔ [D1]: читання/запис документів (SMB)
[P0] ↔ [D2]: SQL-запити, дані додатків

Межі довіри:
Межа 1: Інтернет | Периметр мережі компанії (маршрутизатор Mikrotik) — розділяє зовнішні сутності (E2, E3) та внутрішню мережу
Межа 2: Корпоративна мережа | Хмарні сервіси (GitHub, AWS) — розділяє локальну інфраструктуру та зовнішні хмарні платформи (E4, E5)
Порада У реальному звіті цей текстовий опис доповнюється графічною DFD-діаграмою, побудованою в draw.io. Збережіть діаграму як PNG або PDF та вставте у звіт.

3. Таблиця загроз STRIDE

Для елементів DFD ідентифіковано 10 загроз, що покривають усі 6 категорій STRIDE:

№ Елемент DFD Категорія STRIDE Опис загрози Серйозність
T01 [P0] ІКС «TechSoft» Spoofing Зловмисник надсилає фішинговий лист від імені керівництва або ІТ-відділу з метою отримання облікових даних Google Workspace працівника Висока
T02 [P0] ІКС «TechSoft» Spoofing Підбір пароля (brute-force) до облікового запису VPN-шлюзу для несанкціонованого підключення до корпоративної мережі Висока
T03 Потік: [E3] Клієнт → [P0] Tampering SQL-ін'єкція через форми введення веб-додатку, що дозволяє змінити або видалити дані в базі PostgreSQL Висока
T04 [E4] GitHub Tampering Внесення шкідливого коду у репозиторій через компрометований обліковий запис розробника на GitHub (відсутність обов'язкового code review) Висока
T05 Потік: [E1] Працівник → [P0] Repudiation Працівник надсилає конфіденційний вихідний код конкуренту через особисту пошту та заперечує цей факт через відсутність детального аудиту дій Середня
T06 [D2] БД PostgreSQL Information Disclosure Витік бази даних клієнтів через незахищений бекап у AWS S3 (публічно доступний bucket або скомпрометовані AWS-ключі) Висока
T07 [D1] Файловий сервер Information Disclosure Несанкціонований доступ до фінансових документів та договорів через надмірні права доступу на файловому сервері Середня
T08 Потік: [E3] Клієнт → [P0] Denial of Service DDoS-атака на публічний веб-додаток компанії, що робить його недоступним для клієнтів Середня
T09 [D1] Файловий сервер Denial of Service Шифрування файлів на файловому сервері програмою-вимагачем (ransomware), що поширилася з робочої станції працівника Висока
T10 [P0] ІКС «TechSoft» Elevation of Privilege Використання вразливості в Active Directory або ОС сервера для підвищення прав звичайного користувача до адміністратора домену Висока
Покриття STRIDE У таблиці покрито всі 6 категорій: Spoofing (T01, T02), Tampering (T03, T04), Repudiation (T05), Information Disclosure (T06, T07), Denial of Service (T08, T09), Elevation of Privilege (T10). Переконайтеся, що у вашій роботі також представлено всі категорії.

4. Модель порушника

Характеристика Зовнішній порушник (хакер / кіберзлочинна група) Внутрішній порушник (працівник компанії)
Мотивація Фінансова вигода (ransomware, продаж викрадених даних на darknet), замовлення конкурента, хактивізм Помста (конфлікт, звільнення), фінансова вигода (продаж коду конкуренту), необережність (випадковий витік), соціальна інженерія (жертва фішингу)
Рівень знань та можливостей Середній-високий: використання відомих вразливостей, готових exploit-ів та інструментів (Metasploit, Cobalt Strike), фішингових кампаній Низький-середній: знання внутрішніх систем компанії, використання штатних засобів доступу, можливість копіювання даних на зовнішні носії
Початковий доступ Зовнішній: Інтернет, публічні сервіси (веб-додаток, VPN-шлюз, електронна пошта працівників) Внутрішній: робоча станція в корпоративній мережі, обліковий запис AD, доступ до Git, Google Workspace, файлового сервера
Типові сценарії атак Фішинг → компрометація облікового запису → доступ до внутрішніх ресурсів; експлуатація вразливості веб-додатку (SQLi); DDoS; ransomware через шкідливий вкладений файл Копіювання вихідного коду перед звільненням; випадкове завантаження malware з фішинг-листа; надсилання конфіденційних документів на особисту пошту; використання чужого облікового запису

5. Контрзаходи для топ-3 загроз

Загроза T01: Фішингова атака (Серйозність: Висока)

Тип контрзаходу Захід Як знижує ризик
Організаційний Щоквартальне навчання працівників з розпізнавання фішингу та проведення імітованих фішинг-кампаній (GoPhish) Підвищує пильність працівників, знижує ймовірність того, що хтось натисне на шкідливе посилання або відкриє небезпечний вкладений файл
Технічний Впровадження апаратних ключів двофакторної автентифікації (FIDO2/YubiKey) для Google Workspace та GitHub Навіть якщо працівник введе пароль на фішинговому сайті, зловмисник не зможе увійти без фізичного ключа, прив'язаного до легітимного домену

Загроза T03: SQL-ін'єкція у веб-додатку (Серйозність: Висока)

Тип контрзаходу Захід Як знижує ризик
Організаційний Обов'язковий code review кожного pull request з увагою до безпеки, використання SAST-інструменту (Semgrep) у CI/CD pipeline Виявляє вразливий код до його потрапляння у продакшн; формує культуру безпечного програмування в команді
Технічний Використання виключно параметризованих запитів (Prepared Statements / ORM) та розгортання Web Application Firewall (AWS WAF) Параметризовані запити унеможливлюють впровадження SQL-коду через введення користувача; WAF блокує типові патерни SQL-ін'єкцій на рівні мережі

Загроза T09: Ransomware на файловому сервері (Серйозність: Висока)

Тип контрзаходу Захід Як знижує ризик
Організаційний Розробка та тестування плану реагування на інциденти (incident response playbook) для випадку ransomware: ізоляція, відновлення, повідомлення CERT-UA Зменшує час реагування на інцидент та мінімізує збитки; регулярне тестування плану гарантує його працездатність у критичний момент
Технічний Резервне копіювання за правилом 3-2-1 (3 копії, 2 типи носіїв, 1 offsite) із використанням immutable storage (AWS S3 Object Lock) Навіть при повному шифруванні файлового сервера ransomware, незмінні (immutable) бекапи дозволяють відновити всі дані без сплати викупу

6. Висновок прикладу

Висновок

У ході аналізу ІКС компанії «TechSoft» побудовано контекстну DFD-діаграму з 5 зовнішніми сутностями, 2 сховищами даних та 2 межами довіри. Ідентифіковано 10 загроз за методологією STRIDE з покриттям усіх 6 категорій.

Найнебезпечнішими категоріями STRIDE для IT-компанії є Spoofing (фішинг — найпоширеніший вектор атак) та Information Disclosure (витік вихідного коду та даних клієнтів може спричинити значні репутаційні та фінансові збитки). Зовнішній порушник становить більшу загрозу за частотою атак, проте внутрішній порушник потенційно небезпечніший через наявний доступ до критичних систем.

Ключові рекомендації: впровадити апаратну двофакторну автентифікацію (FIDO2) для захисту від фішингу, забезпечити безпечну розробку через code review та параметризовані запити, а також налагодити надійне резервне копіювання для протидії ransomware.

Вимоги до звіту

Звіт має містити такі розділи (загальний обсяг — 3–4 сторінки):

  1. Титульна сторінка — назва дисципліни, номер та назва практичної роботи, ПІБ студента, група, дата
  2. Мета роботи
  3. Короткий опис організації — 5–7 речень
  4. DFD-діаграма — скріншот з draw.io та текстовий опис елементів
  5. Таблиця загроз STRIDE — 8–10 загроз із покриттям усіх 6 категорій
  6. Модель порушника — 2 типи порушників (зовнішній та внутрішній) із характеристиками
  7. Контрзаходи для топ-3 загроз — по 2 контрзаходи (організаційний + технічний) для кожної
  8. Відповіді на контрольні питання
  9. Висновки
Важливо DFD-діаграма має бути побудована у draw.io та вставлена у звіт як зображення (скріншот). Діаграми, намальовані від руки, не приймаються.

Контрольні питання

  1. Що таке DFD і які основні елементи вона містить? Для чого використовується DFD при моделюванні загроз?
  2. Розкрийте кожну категорію методології STRIDE та наведіть приклад загрози для кожної категорії.
  3. Що таке межа довіри (trust boundary)? Чому загрози найчастіше виникають саме на межах довіри?
  4. Опишіть різницю між зовнішнім та внутрішнім порушником. Який із них, на вашу думку, становить більшу загрозу і чому?
  5. Як співвідносяться категорії STRIDE з тріадою CIA (Confidentiality, Integrity, Availability)?
